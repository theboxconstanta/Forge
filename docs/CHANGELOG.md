## 2026-07-20 — Online Payments (Stripe), Phase 5a: RPC authorization layer shipped
Groundwork for member-initiated Stripe Checkout, built on top of the frozen Financial Domain — no Stripe SDK, Checkout Session, or webhook Edge Function exists yet; this phase is purely the RPC-layer trust and authorization model those future pieces will call into. `create_order_for_subscription`, `register_payment`, `create_subscription`, `activate_queued_subscription` each gained a `service_role`-authorized path (tenant resolved from the target row, since `auth.uid()`/`my_gym_id()` are null under that role); `register_payment` gained idempotent retry on `(provider, provider_reference)`; `create_subscription` gained a self-service path that never accepts a caller-attested payment amount and always creates a pending Order immediately; `activate_queued_subscription` gained a paid-Order activation guard blocking any non-admin caller (member or `service_role`) from activating before the Order is `status='paid'`, with admin retaining an unconditional override and the no-Order admin-scheduled-renewal path left untouched. A pre-existing, unrelated defect was found by this phase's mandated validation (not introduced by it): `subscriptions_restrict_member_update()`, part of the waitlist auto-booking system, rejected every `service_role` write to `subscriptions` — fixed with one added bypass clause, every other caller class re-verified unaffected. Two validations were mandatory before sign-off, both passed: idempotency (same webhook/`provider_reference` delivered twice → exactly one Payment, one Order transition to `paid`, one active Subscription) and concurrency (two genuinely lock-forced-parallel sessions on the same order/subscription → same one-each outcome, confirmed via `clock_timestamp()` proving real blocking, not sequential luck). 5 migrations (`20260720130100`–`20260720130500`), all applied and verified live. Full reasoning: `/docs/DECISIONS.md`.

## 2026-07-20 — Financial Domain shipped, closed, and frozen
Regex-parsed `subscriptions.notes` revenue tracking replaced end-to-end with a structured `orders`/`payments` model (`Subscription → Order → Payment (method/provider/provider_reference) → Refund → Reporting`). Five phases: schema (`orders`, `payments`, constraints, triggers, RLS) → core RPCs (`create_order_for_subscription`, `register_payment`, `refund_payment`) → subscription-lifecycle RPCs (`create_subscription`, `activate_queued_subscription`, `delete_queued_subscription`, `end_subscription` — added after a gap analysis found no original RPC actually wrote to `subscriptions`) → application cutover (`saveAbonament`, `adminActiveazaAboQueued`, `activateQueuedSubscription`, `stergeAbonament` rewired to call the RPCs exclusively, legacy `notes` write removed) → reporting migration (`fetchRapoarte`'s revenue figure now anchored on successful `payments` events, not subscription/order creation dates). One post-closure extension: Payment Methods — `payments.method` canonicalized to a closed vocabulary (`cash`/`card`/`bank_transfer`/`comp`, no `'other'`), completing a capability the original schema already had but no code path used; `create_order_for_subscription` widened to admin-or-owner so member self-service activation produces an Order (`register_payment` deliberately left admin-only — see `/docs/DECISIONS.md`). 27 migrations, all committed and verified live. Full ADR record and reasoning: `docs/2026-07-20_Financial_Domain_Architecture_Working_Session.md` (ADR-001 through ADR-013). Every phase validated against real production data (rolled back or explicitly cleaned up), including a genuine end-to-end browser walkthrough for the final phase. Two real bugs caught and fixed before sign-off: `CREATE OR REPLACE FUNCTION` silently creating a duplicate overload instead of replacing when a parameter signature changed arity, and a dropped grants block briefly leaving three RPCs callable by `anon` — both found by checking live state rather than trusting migration output.

## 2026-07-18 — Isolated demo environment shipped
Built a fully separate Supabase project (`forge-demo`) + Vercel deployment for external product/UX review, seeded with realistic fictional gym data. Schema-only `supabase db dump` from production applied to the new project; discovered and fixed a real gap in that process — `on_auth_user_created` lives in the `auth` schema and isn't captured by a `--schema public` dump, so it had to be recreated by hand (see `PROJECT_STATE.md` §6). Production was never written to (link + dump only, read-only). See `scripts/seed-forge-demo*.mjs` for the reusable seed script.

## 2026-07-17/18 — Workout Composer: formatId elimination audit
Full audit of the Composer + rendering pipeline confirmed zero remaining presentation decisions based on literal workout-format identity — the 3 that existed (RFT/For Time/Partner WOD "repeated rounds"; EMOM/Tabata/Intervals/Death By archetype word; Death By's score note) were all replaced with checks on fields the catalog already had, with zero new catalog metadata. Validated live (before/after byte-identical output on real production WODs) and via 6 new regression tests (349 total).

## 2026-07-17/18 — Workout Composer shipped
`workoutComposer.js` (pure `WorkoutSection → ComposedWorkout` transform) + `ComposedWorkoutView.jsx` (React rendering, role/weight-driven) implemented, tested, and wired into a live preview in the Admin WOD editor. Validated end-to-end against 17 real production WODs; one real rendering bug found and fixed (single-movement sections duplicating the movement name as heading and body line). A live before/after test proved the "structured data in → correct whiteboard layout out, zero code changes" hypothesis on a real legacy WOD.
